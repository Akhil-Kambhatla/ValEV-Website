# Accessibility Findings — Automated Pass

Generated by running `npm run a11y` (`a11y/axe-audit.mjs`) against all 10 routes with axe-core 4.x
(WCAG 2.0/2.1 A + AA rule sets), plus two targeted manual checks the code-only audit in `AUDIT.md`
§6 flagged as unverified. No application code was changed to produce this report.

Routes covered: `/`, `/about`, `/technology`, `/contact`, `/partner`, `/partner/franchise`,
`/partner/host`, `/partner/fleet`, `/privacy`, `/terms`. (`/network` is not a standalone route —
`NetworkSection` is embedded in `/`.)

---

## 1. Confirmed violations

### 1.1 `<select>` elements with no accessible name — `/partner/franchise`
- **WCAG**: 4.1.2 Name, Role, Value (A); 1.3.1 Info and Relationships (A)
- **Rule**: axe `select-name` (critical impact)
- **Where**: `components/FocoCalculator.tsx`, `SlotSelect` (~lines 53–109), rendered 3× on
  `/partner/franchise` via `FocoCalculator` (line 349 of `PartnerFranchisePage.tsx`) as "Machine 1,"
  "Machine 2," "Machine 3."
- **Cause**: the `<label>` wraps plain text but has no `htmlFor`, and the sibling `<select>` has no
  `id` — so the label is visually adjacent but not programmatically associated with the control.
  Screen reader users landing on the `<select>` hear only "combo box," not "Machine 1."
- **Fix shape** (not applied this session): give each rendered `<select>` a stable `id` (e.g.
  `foco-slot-1`) and add matching `htmlFor` on the label. Purely structural — no visual or copy
  change, doesn't touch frozen theme values.

This was the only axe **violation** (as opposed to "incomplete") found on any route.

## 2. Needs-human-judgment (axe inconclusive, but not blocking)

### 2.1 Navbar link contrast — reported "incomplete" on all 10 routes
- **Rule**: axe `color-contrast`, `bgGradient` message ("Element's background color could not be
  determined due to a background gradient")
- **Where**: the 5 primary nav links (`About`, `Network`, `Technology`, `Partner`, `Contact`) in
  `components/Navbar.tsx`, which sit above the hero's `aria-hidden` radial-gradient ambient veil
  (`z-index: 5`) when the page is unscrolled (`components/Navbar.tsx:93-105` — the header has no
  background of its own until `scrolled` becomes true).
- **Why axe can't resolve it**: axe's contrast engine cannot sample a computed pixel color through a
  CSS gradient background — this is a known tool limitation, not evidence of a real failure.
- **Manual verification done this session**: nav text is `--silver-mid` (#C4C8CE). Worst case for
  contrast is if the ambient gradient is fully transparent at the navbar's position, leaving only
  the page base `--bg-hero` (#07080A) behind it. Computed WCAG contrast ratio of `#C4C8CE` on
  `#07080A` = **~11.9:1**, comfortably above both the 4.5:1 AA and 7:1 AAA thresholds for normal
  text. Since the gradient overlay, where present, only ever adds *more* darkness (up to
  `rgba(7,8,10,0.90)`), the real on-screen contrast can only be equal to or higher than this
  worst case. Once scrolled, the navbar gets its own `rgba(7,8,10,0.82)` backdrop-blurred panel,
  which is equivalently dark.
- **Verdict**: effectively clean; flagged as "incomplete" by tooling only, not a real defect.
  Bucketed here rather than "confirmed clean" because it rests on a manual luminance calculation
  rather than a tool-verified pixel sample — a two-second visual scroll-through would fully close it
  if you want zero residual doubt.

No other `color-contrast` incompletes were found — this was the same single pattern repeating
across every route (it's the same navbar).

## 3. Targeted verification (the two items AUDIT.md §6/§8.4 flagged as unconfirmed)

### 3.1 `--silver-lo` (#9AA0A8) small-text contrast — **confirmed clean**
Checked every element on every route whose computed text color is exactly `--silver-lo`
(eyebrow labels, calculator disclaimers, footer sub-links, etc. — 3 to 18 elements per route,
98 element-instances total across the site), walking up the DOM to each element's actual
rendered (opaque) background and computing the real WCAG contrast ratio, including the correct
"large text" 3:1 threshold where font-size/weight qualifies.

**Result: 98/98 pass WCAG AA.** Every `--silver-lo` instance sits on a background at or near the
site's `#07080A`–`#242E3E` card range, and #9AA0A8 clears 4.5:1 against all of them with margin.
No violations, no borderline cases.

### 3.2 Range slider (`<input type="range">`) touch-target size — **1 confirmed issue, plus a scope correction**

**Scope correction first**: AUDIT.md's phrasing ("both live calculators") assumed
`HostEarningsCalculator` (`components/HostEarningsCalculator.tsx`) is live. It is not — grepping the
whole repo, that component is defined but **never imported or rendered anywhere**. It's dead code,
not reachable via any route, so its two range sliders aren't part of the site's actual accessibility
surface. (Flagging this as a separate incidental finding — not fixing it, since removing/wiring up
dead code is a scope decision outside this accessibility pass.)

`FocoCalculator` (`/partner/franchise`) has no range slider — it uses `<select>` dropdowns (see §1.1).

That leaves exactly **one** live range slider on the whole site:

- **`/partner/fleet`** — `FleetSavingsCalc`'s "Number of vehicles" slider
  (`components/PartnerFleetPage.tsx:175-187`, `id="fleet-vehicles"`)
- **Measured at a 375×812 mobile viewport**: rendered hit-area **335×16px** (`cssHeight: 16px`,
  no padding).
- **WCAG**: 2.5.8 Target Size (Minimum) (AA, WCAG 2.2) recommends **24×24 CSS px** minimum for
  pointer targets (with some exceptions that don't clearly apply to a primary interactive control
  like this). At 16px tall, this is **9px under** that guideline in the touch-critical dimension.
  Width (335px) is not a concern — a full-width slider track is expected to be wide.
- **Practical note**: native `<input type="range">` in mobile Safari/Chrome generally makes the
  *entire track* draggable (not just the visible thumb), so the slider isn't unusable — but the
  visually/programmatically defined hit-area is still short of the 24px guideline, and a thin track
  is harder to acquire precisely with a thumb on a touch screen, especially for the two other
  read-only calculators next to it that use tap-target pill buttons at compliant sizes.
- **Fix shape** (not applied this session): increasing the slider's effective touch height (e.g. via
  `padding-block` on the input, or a taller custom thumb) is a layout/sizing change, not a color
  change — doesn't touch frozen theme values, should be low-risk to apply in a follow-up session if
  desired.

## 6. Re-running this audit

```
npm run a11y
```

Starts (or reuses) a `next dev` server on port 4173, runs axe against all 10 routes at a 1440×900
desktop viewport, and separately checks range-slider hit-area sizes at a 375×812 mobile viewport.
Set `BASE_URL` to point it at an already-running server instead (e.g. `BASE_URL=http://localhost:3000
npm run a11y`). Exits non-zero if axe reports any violations, so it's CI-friendable later if wanted.
